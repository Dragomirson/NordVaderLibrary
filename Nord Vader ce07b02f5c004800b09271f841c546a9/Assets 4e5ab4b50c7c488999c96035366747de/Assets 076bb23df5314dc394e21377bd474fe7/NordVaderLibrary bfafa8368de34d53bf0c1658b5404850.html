Overview:

The NordVaderLibrary Plugin is a comprehensive toolkit designed for Unreal Engine developers to enhance their workflow with a variety of powerful, asynchronous functions. These functions are specifically crafted to handle complex, resource-intensive tasks without interrupting the main game thread, ensuring smooth and efficient performance.

Features:

  • Asynchronous Actor Spawning: Efficiently spawn actors, emitters, and decals without causing game lag.
  • Parallel Processing: Utilize multi-threading for intensive calculations and checks.
  • Soft Reference Handling: Manage and load assets dynamically using soft references to optimize memory usage.
  • Advanced Damage Handling: Apply damage with precision using safe and parallel methods.
  • Math Utilities: Perform complex range checks and random value generation effortlessly.

Core Functions:

Functions with the Task suffix and without it in the NordVaderLibrary Plugin differ based on the following criteria:

Functions without the Task suffix:

  1. Synchronous Execution:
    • These functions run in the main game thread.
    • They block further code execution until they complete.
    • This can cause delays in the game if tasks require significant resources or time.
  1. Ease of Use:
    • These functions are easier to debug and use since all operations occur in a single thread.
    • Suitable for simple scenarios where tasks are quick and don't require significant resources.
  1. Direct Use of Soft References:
    • These functions can load classes or assets via soft references and perform necessary actions synchronously.
    • This allows for dynamic resource loading without permanently storing them in memory, reducing overall memory usage.

Functions with the Task suffix:

  1. Asynchronous Execution:
    • These functions run in separate threads, allowing the main game thread to continue without delays.
    • They enhance performance by distributing the load across multiple threads.
  1. Use in Resource-Intensive Tasks:
    • Suitable for tasks requiring significant resources or time, such as asset loading or complex computations.
    • Asynchronous execution allows the main game to continue, minimizing delays and boosting overall performance.
  1. Complexity of Implementation:
    • They use mechanisms like AsyncTask and delegates to notify when a task is complete.
    • Can also use soft references for dynamic resource loading, ensuring efficient memory use without permanently storing loaded data.

Examples:

  • AsyncSpawnActor (without the Task suffix):
    • Asynchronously spawns an actor in the game world.
    • Executes synchronously in the main game thread.
  • AsyncSpawnActorTask (with the Task suffix):
    • Queues actor spawning as an asynchronous task.
    • Runs in a separate thread, not blocking the main game thread.

Thus, the primary difference between these functions is in their execution mode: synchronously in the main thread or asynchronously in separate threads, affecting performance and suitable use cases.

Actor Management:

  • AsyncSpawnActorTask: Queues actor spawning as an asynchronous task.
  • SetSeparationWeight: Adjusts the separation weight for crowd following components to manage AI behavior dynamically.
  • AsyncSpawnActor: Asynchronously spawns an actor in the game world.
  • AsyncDestroyActorTask: Asynchronously destroys an actor and calls a delegate upon completion.
  • AsyncPlayAnimationTask: Asynchronous task for playing an animation on a skeletal mesh component.
  • AsyncPlayAnimation: Asynchronously plays an animation on a skeletal mesh component.
  • AsyncSetAnimInstanceClassTask: Asynchronous task for setting an animation instance class on a skeletal mesh component.
  • AsyncSetAnimInstanceClass: Asynchronously sets an animation instance class on a skeletal mesh component.

Load Management:

  • AsyncLoadStaticMeshTask: Asynchronous task for loading a static mesh.
  • AsyncLoadSkeletalMeshTask: Asynchronous task for loading a skeletal mesh

Emitter and System Management:

  • AsyncSpawnEmitter: Spawns particle systems asynchronously.
  • AsyncSpawnEmitterTask: Manages particle system spawning as an asynchronous task.
  • AsyncSpawnSystemAttached: Attaches and spawns Niagara systems asynchronously.
  • AsyncSpawnSystemAttachedTask: Queues Niagara system spawning as an asynchronous task.
  • AsyncSpawnSystemAtLocationTask: Asynchronous task for spawning a particle system at a specific location.

Decal Management:

  • AsyncSpawnDecalAtLocation: Spawns decals at specified locations asynchronously.
  • AsyncSpawnDecalAtLocationTask: Manages decal spawning as an asynchronous task.

Sound Management:

  • AsyncPlaySoundAtLocation: Plays sounds at specific locations asynchronously.
  • AsyncPlaySoundAtLocationTask: Manages sound playback as an asynchronous task.

Damage Management:

  • ApplyDamageSafely: Safely applies damage to an actor using soft references.
  • ApplyDamageActorsTask: Manages applying damage to multiple actors as an asynchronous task.
  • ApplyDamageActorTask: Queues damage application to a single actor as an asynchronous task.
  • ApplyPointDamageActorTask: Manages point damage application to actors asynchronously.
  • ApplyRadialDamageActorTask: Applies radial damage asynchronously.
  • ApplyDamageToActors: Applies specified damage to a list of actors.

Utilities:

  • InRange: Checks if a value is within a specified range.
  • InRange_Pure: Pure version of InRange for simplified checks.
  • RandomFloatInRange: Generates random values within a specified range.
  • RandomFloatInRange_Pure: Pure version for generating random values.
  • GetBoolByChance: Determines if an event should occur based on a given chance.
  • GeneratePointsOnSphereFibonacciParallel: Generates points on a sphere using the Fibonacci lattice method in parallel.
  • PerformFibonacciLatticeTracesParallel: Performs ray traces from the center in directions based on points generated by the Fibonacci lattice method and returns an array of hit results.
  • GetMemoryUsage: Retrieves the current memory usage statistics of the system.

Parallel Processing:

  • GetActorsBySoftClassParallel: Filters actors by class using parallel processing.
  • GetWasRecentlyRenderedbyActorsParallel: Checks if actors were recently rendered using parallel processing.
  • ParallelSortActorsByDistance: Sorts actors by distance to a point using multi-threading.
  • CheckActorsInFOVParallel: Asynchronously checks if actors are within a specified Field of View (FOV) angle of the camera and within a given distance. Optionally verifies if actors are currently being rendered.
  • TraceSocketsInActorParallel: Asynchronously performs traces from a specified start location to sockets on skeletal mesh components tagged with a specified name. It returns a boolean indicating if any trace was successful.
